# 改善提案

対象: Viewpoint（html_maker）
テーマ: ① 「選択して編集できない部分」の解消 ② 既存 HTML（例: `三者比較/` の各 HTML）の読み込み・編集対応

---

## 1. 現状調査: 選択・編集できない箇所の棚卸し

`Renderer.tsx`（クリック → Selection 発行）と `Inspector.tsx`（Selection → 編集 UI）を突き合わせた結果、以下のギャップがあります。

### 1-1. 【重大】sentence ブロック本体が選択できない

- `Renderer.tsx` の `case "sentence"`（186行目付近）では `<p className="r-sentence">` に **onClick がなく**、各 token にしか onClick がない。
- そのため `{ kind: "block" }` の Selection を作る手段が UI 上に存在せず、
  - ブロックの **移動・削除**（BlockToolbar）ができない
  - `SentenceEditor`（語の追加・削除・一括編集）に**到達できない**
  - ブロック単位の **AI 編集**ができない
- ※ ブロック追加直後だけ `onAddBlock` が block 選択にするので一瞬編集できるが、一度選択を外すと二度と戻れない。

**対処**: `<p className="r-sentence">` に他ブロックと同じ `selectBlock` ハンドラを付ける（token クリックは `stopPropagation` 済みなので共存可能）。token 以外の余白クリックでブロック選択になる。

### 1-2. 【重大】tree ブロック本体が選択できない

- `case "tree"` も root ノードと枝しかクリックできず、ブロック自体の Selection を作れない。
- 結果、樹形図の **移動・削除ができない**。

**対処**: `.r-tree` のコンテナに `selectBlock` を付ける（root/枝クリックは `stopPropagation` 済み）。

### 1-3. 【重大】token / branch / tree-root 選択時にブロック操作へ戻れない

- `BlockToolbar`（上下移動・削除）は `selection.kind === "block"` のときしか表示されない（`Inspector.tsx` 127行目）。
- 語や枝を選んだ状態から「親ブロックを選択」する導線がない。

**対処**: Inspector に常設の「親ブロックを選択」ボタンを置く、または全 Selection 種別で BlockToolbar を表示する（削除対象が「ブロック全体」であることを明記）。

### 1-4. 【重大】table のセルが編集できない（JSON 直書きのみ）

- セル・行・列は選択不可。編集手段は Inspector の「行 JSON」textarea のみで、教師向け UX として厳しい。
- セルには `data-vp-field` がないため **範囲選択マーカーも付けられない**。table の `title` も同様。

**対処（2段階）**:
1. Inspector の TableEditor をグリッド UI 化: セルごとに `<input>`、「＋行」「−行」「＋列」「−列」ボタン。JSON textarea は畳んで残す。
2. Selection に `{ kind: "table-cell"; blockId; row; col }` を追加し、ビュー上のセルを直接クリック → Inspector で該当セルを編集できるようにする。

### 1-5. 【中】analysisCard の items が JSON 直書きのみ

- `items` の編集は「項目 JSON」textarea のみ。item 個別の選択も不可。
- `dt`（ラベル側）と `tag` には `data-vp-field` がなく、範囲マーカー対象外（`dd` 値側のみ対応済み）。

**対処**: items を行リスト UI 化（label / value / role セレクト / 削除 / ＋追加）。余力があれば Selection に `{ kind: "analysis-item" }` を追加してビューから直接選択可能にする。

### 1-6. 【中】AnalysisCardEditor / TableEditor の state 同期バグ

- 両エディタは `useState(JSON.stringify(block.items…))` で**初回マウント時の値を保持**するが、コンポーネントに `key` がないため、
  - 別のカード/表を選択しても **前のブロックの JSON が表示されたまま**になる
  - AI 編集や undo でブロックが差し替わっても textarea が追従しない
- 「編集できない」体感の一因。

**対処**: `Inspector.tsx` で `<AnalysisCardEditor key={block.id} …>` / `<TableEditor key={block.id} …>` と key を付ける。さらに undo/AI 反映に追従させるなら「block が外部変更されたら draft をリセットする」`useEffect` を追加。

### 1-7. 【小】範囲選択マーカーの対象フィールドが限定的

`data-vp-field` が付いているのは heading/paragraph の text、analysisCard の title/source/quote/item値/takeaway、note の body のみ。以下は範囲選択不可:

- table のセル・タイトル
- note の label、image の alt/figcaption
- analysisCard の tag / dt ラベル

**対処**: 型上 `marks` を持てるブロックを広げるより、まず頻度の高い **table セル**への対応を優先（1-4 のセル選択と同時に実装）。

### 1-8. 【小】その他

- **既存マーカーの一覧・個別削除 UI がない**（範囲を選び直して「解除」するしかない）→ Inspector に該当ブロックのマーカー一覧＋削除ボタンを追加。
- **役割パレットが「未選択時」しか出ない** → Legend（凡例チップ）クリックでパレット編集を開けるようにする、または Inspector にタブ化して常時アクセス可能に。
- **doc タイトル**はマーカー・AI 編集の対象外（現状は topbar の input のみ）。優先度低。

## 2. 既存 HTML の読み込み（インポート）機能

「`三者比較/` や `~/Desktop/提言/03_index-edited.html` のような、Claude / Gemini などで既に作った単発 HTML を取り込んで、このエディタで編集したい」を実現します。HTML の由来ごとに性質が違うため、**3 層構成**を提案します。

### 2-0. 実例分析: `提言/03_index-edited.html`

ユーザー提示の実例を解析した結果:

- 固定ナビ、グラデーションのヒーロー、11 の `<section>`、カードグリッド×16、統計カード×7、タイムライン、Do/Don't 比較ボックス、FAQ（`<details>`）、表×2 という**ランディングページ型の提言書**。
- 内容（テキスト）は LessonDoc の語彙（heading / paragraph / analysisCard / table / note）へ概ね写像できるが、**デザイン（配色・グリッド・ヒーロー等）は写像できない**。Viewpoint 形式に変換すると見た目は Viewpoint 標準スタイルに置き換わる。
- つまりインポートには2つの目的があり、要件が異なる:
  - **(a) 元のデザインを保ったまま内容を編集したい** → セクション単位の raw ブロック取り込み＋元 CSS の保持（下記 2-3'）
  - **(b) Viewpoint の構造化教材として作り直したい** → AI インポート（2-2）

このファイルのようなデザイン重視の文書は (a) が本命になるため、**raw ブロック運用を「逃げ道」から「第一級のインポート経路」に格上げ**する必要があります。具体的には:

1. `<body>` 直下の `<section>` / `<header>` / `<nav>` 等を**1 セクション = 1 raw ブロック**として分割取り込み → ブロック単位の並べ替え・削除・AI 編集（既存 `/api/edit`）がそのまま効く。
2. 元の `<style>` を保持する場所として `LessonDoc` に任意フィールド **`customCss?: string`** を追加（`types.ts` / `validate.ts` / `exportHtml.ts` に数行ずつ）。エクスポート時に標準スタイルへ追記して出力すれば、**見た目をほぼ完全に維持したまま往復**できる。
3. raw ブロックの編集は現状 textarea の HTML 直書きのみなので、(a) 経路の実用性はブロック単位 AI 編集（「この統計カードの数値を変えて」等）が担う。`SCHEMA_BLOCK` プロンプトに「raw ブロックは HTML の構造・class を保って部分修正する」旨の指示を追加する。

### 2-1. 【必須・低コスト】自アプリ書き出し HTML のラウンドトリップ対応

現在の `exportHtml.ts` は見た目の HTML だけを出力しており、**書き出した HTML から LessonDoc を復元できません**。まずここを塞ぎます。

- `exportHtml()` の `<head>` 内に元データを埋め込む:
  ```html
  <script type="application/json" id="viewpoint-doc">
  {…LessonDoc の JSON（</script> 対策で "<" を < にエスケープ）…}
  </script>
  ```
- インポート時はまずこの script タグを探し、あれば `parseLessonDoc()`（`lib/validate.ts`）に通して**無劣化で復元**。
- これにより「書き出し HTML が配布物であり保存形式でもある」状態になり、localStorage が消えても教材を回収できる。

### 2-2. 【推奨】任意 HTML の AI インポート

`gemini canvas.html` のような他ツール製 HTML は構造がバラバラなので、決定的パースでは限界があります。既存の AI 基盤（`lib/aiServer.ts` / `/api/generate`）を流用して変換します。

- 新 API: `app/api/import/route.ts`
  - 入力: `{ provider, apiKey, model, html }`
  - 前処理（サーバー側）: `<script>`・`<style>`・コメントを除去しつつ、**class 名やインライン color は残す**（色分け＝意味の手がかりになるため）。トークン節約に body 内のみ送る。
  - プロンプト: `lib/prompts.ts` に `SCHEMA_IMPORT` を追加。`SCHEMA_DOC` の型定義を再利用し、
    - 「この HTML は完成済み教材。**内容を変えず**に LessonDoc へ構造化せよ」
    - 「色分けされた語句 → rolePalette + role / marks に対応付けよ」
    - 「表 → table、カード状の枠 → analysisCard、コールアウト → note」
    - 「どうしても表現できない装飾部分のみ raw ブロックに退避してよい」
  - 既存の `/api/generate` と同じ SSE ストリーミング＋修復ロジック（malformed JSON 修復は実装済み）を流用。
- 生成後は通常の LessonDoc として保存 → エディタへ遷移。以降はブロック単位で編集可能。

### 2-3. 【推奨】決定的パーサ＝セクション分割インポート（API キー不要）

`lib/importHtml.ts` を新設し、ブラウザの `DOMParser` で変換する。2-0 の方針に従い、**デザイン維持モード（a）と構造化モード（b）の 2 モード**を持たせる:

**(a) デザイン維持モード**（`提言/03_index-edited.html` のようなデザイン重視の文書向け・デフォルト）

- `<body>` 直下のセクション単位で分割し、各セクションを raw ブロックに格納
- `<style>` の中身を `doc.customCss` に保存（2-0 参照）
- `<title>` / 最初の `h1` を doc.title に

**(b) 構造化モード**（シンプルな文書向け）

| HTML | 変換先 |
|---|---|
| `h1`〜`h3` | heading |
| `p` / `li` | paragraph |
| `table` | table（thead → columns、tbody → rows） |
| `img` | image |
| `blockquote` / `<details>` / コールアウト風 div | note |
| 変換できない要素 | raw ブロック（outerHTML をそのまま保持） |

いずれのモードでも「とりあえず取り込んで、raw ブロックをブロック単位 AI 編集（既存の `/api/edit`）で修正・順次構造化する」という運用ができる。

### 2-4. UI 追加

- **ホーム（`app/page.tsx`）**に「📂 HTML を読み込む」ボタンを追加 → `app/import/page.tsx` へ。
- インポートページの流れ:
  1. `.html` ファイル選択（またはドラッグ＆ドロップ / 貼り付け textarea）
  2. 自動判定: 埋め込み JSON あり → 即復元（2-1）／なし → 「デザイン維持で取り込む」「簡易構造化」「AI で構造化」から選択（2-3a / 2-3b / 2-2）
  3. 変換結果をプレビュー → 「教材として保存」で `saveLesson()` → エディタへ
- `new/page.tsx` の PDF アップロードと同様、ファイルサイズ上限（〜3MB）とオンラインのみファイルのガードを流用。

### 2-5. セキュリティ注意

外部 HTML を raw ブロックに入れると `Renderer.tsx` の `dangerouslySetInnerHTML` でそのまま実行されます。インポート経路を作るなら:

- インポート時に `script` タグ・`on*` 属性・`javascript:` URL を除去（サニタイズ）してから raw に格納する
- 依存を増やしてよければ `dompurify` の導入が確実

## 3. 実装順序の提案

| 順 | 内容 | 触るファイル | 規模 |
|---|---|---|---|
| 1 | sentence / tree ブロック本体の選択対応（1-1, 1-2） | `Renderer.tsx` | 小 |
| 2 | エディタ state 同期バグ修正（1-6） | `Inspector.tsx` | 小 |
| 3 | 親ブロック選択導線 / BlockToolbar 常時表示（1-3） | `Inspector.tsx` | 小 |
| 4 | 書き出し HTML への JSON 埋め込み（2-1） | `exportHtml.ts` | 小 |
| 5 | インポートページ＋埋め込み JSON 復元＋セクション分割/簡易パーサ＋`customCss` 対応（2-3, 2-4, 2-5） | `app/import/page.tsx`（新規）、`lib/importHtml.ts`（新規）、`app/page.tsx`、`types.ts`、`validate.ts`、`exportHtml.ts` | 中 |
| 6 | table のグリッド編集 UI（1-4 前半） | `Inspector.tsx` | 中 |
| 7 | analysisCard items の行リスト UI（1-5） | `Inspector.tsx` | 中 |
| 8 | AI インポート（2-2） | `app/api/import/route.ts`（新規）、`lib/prompts.ts`、`app/import/page.tsx` | 中 |
| 9 | セル選択（table-cell Selection）＋セルへの範囲マーカー（1-4 後半, 1-7） | `types.ts`、`Renderer.tsx`、`Inspector.tsx`、`exportHtml.ts` | 大 |
| 10 | マーカー一覧管理・パレット常時アクセス（1-8） | `Inspector.tsx` | 小 |

1〜4 は互いに独立で即効性が高く、この 4 つだけで「選択できない」問題の大半（sentence・tree・エディタ不追従）が解消します。5 以降でインポート、6 以降で編集 UX の底上げ、という段取りです。
